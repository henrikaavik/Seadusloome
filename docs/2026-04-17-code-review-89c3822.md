## Findings

No blocking correctness, security, or regression findings. A handful of non-blocking risks below, ordered by severity.

### Chat (`app/chat/`)

**1. RAG retrieval cancellation path misses `retrieval_done` cleanup** — `app/chat/orchestrator.py:650` ff.
A user-initiated `Peata` (CancelledError) that lands while `await retriever.retrieve(...)` is in flight escapes past the local except handlers without emitting `retrieval_done`. The UI can stick on "Otsin konteksti…". Wrap the RAG block in an additional `except asyncio.CancelledError: emit retrieval_done; raise`.

**2. Partial-content persistence runs blocking psycopg I/O from cancellation handlers** — `app/chat/orchestrator.py:919, 895, 940, 952`.
`_persist_partial_assistant` is synchronous and runs inside `except CancelledError/TimeoutError/WebSocketSendTimeout`. Blocks the event loop ~10–50 ms during cancel/timeout. Acceptable today; wrap with `asyncio.to_thread(...)` if latency spikes under contention.

**3. `stopped` event not shielded against double-cancel** — `app/chat/orchestrator.py:927`.
A second cancel (double-click on Peata, socket teardown) during the stopped-event send can drop the final frame. Low probability; nest `asyncio.shield(_safe_send(stopped))`.

**4. `tokens_in`/`tokens_out` never populated on persisted assistant rows** — `app/chat/orchestrator.py:722-728, 767-797, 988-989`.
Pre-existing: streaming loop does not handle a `"usage"` StreamEvent. Cost attribution via `llm_usage` is unaffected, but per-message token metrics are always NULL.

**5. Archive-row empty swap leaves pagination counts stale on `?archived=1`** — `app/chat/handlers.py:386-394`.
The row vanishes but the filter toolbar's count still includes it. Fix with an `HX-Trigger: chat:counts-stale` and a listener refresh. UX nit, not correctness.

**6. `_render_conversation_row` reads back the conv on the same connection that just committed** — `app/chat/handlers.py:324-327, 424-428`.
Correct in psycopg (read-your-writes, session-local) but would silently break on a future read-replica split. Add an inline comment pinning the expectation.

**7. `from_list` form re-read swallows `ClientDisconnect`** — `app/chat/routes.py:1096-1100`.
Delete is idempotent so harmless, but log reads "Failed to read delete form" when the real cause is client gone. Narrow the except.

### Drafts + migration (`app/docs/`, `migrations/021`)

**8. `can_edit_draft` docstring promises "same-org editor" but implementation is owner-only** — `app/docs/retry_handler.py:90-93` vs `app/auth/policy.py:121-131`.
Policy today mirrors `can_delete_draft` (row owner or sys admin). Matches the `test_retry_other_org_returns_not_found` expectation. Either fix the comment to "owner-only" or, if product intent is broader, split out `can_retry_draft` explicitly.

**9. Migration 021 pattern set is narrow** — `migrations/021_mask_leaked_env_errors.sql:64-70`.
Only five env-var names matched. Real error-mapping surface also emits `DATABASE_URL`, `FUSEKI_URL`, `SMTP_*`, `JWT_SECRET`, and stack traces containing `File "/app/...`. Recommend auditing the live DB (`SELECT DISTINCT error_message FROM drafts WHERE status='failed'`) and widening — or replace the blocklist with a positive allowlist targeting any `error_message` not matching the canonical Estonian fallback.

**10. Retry race with queued worker could restore stale `failed` state** — `app/docs/retry_handler.py:65-79`.
Guard on `status='failed'` is correct, but a stuck `parse_draft` job from a prior pipeline could concurrently land an `UPDATE ... status='failed'` after the reset. Mitigate by deleting pending `jobs` rows for the draft in the same txn, or by bumping a generation counter that handlers check before writing.

**11. Retry audit log loses prior error context** — `app/docs/retry_handler.py:170-174`.
`log_action("draft.retry", ...)` is called after `error_message`/`error_debug` have been wiped. Capture `{"prior_error": draft.error_message[:200]}` in the payload before the reset so the audit trail reconstructs what was retried.

**12. "Analüüsitud N h M min S s" label uses `updated_at - created_at`** — `app/docs/routes.py:227-244`.
Any later edit (rename, link-VTK, re-tagging) bumps `updated_at` and retroactively inflates the label on a ready draft. Record a dedicated `processing_completed_at` or read the terminal `jobs.completed_at`. Docstring acknowledges the proxy; the failure mode is real.

**13. Local import from `app.docs.routes` signals missing `_helpers.py`** — `app/docs/retry_handler.py:97`.
`_not_found_page` / `_parse_uuid` belong in a shared module; the current in-function import is a cycle workaround.

### UI / theme

**14. `<meta name="color-scheme">` still advertises `"light dark"`** — `app/main.py:130`.
Should be `"dark"` so the UA paints native controls, scrollbars, and the pre-CSS background dark during FOUC window.

**15. `PageShell.theme` default still `"system"`** — `app/ui/layout/page_shell.py:57`.
Parameter is now ignored but the stale default contradicts the new contract. Either default to `"dark"` or drop the parameter (every call site reads `get_theme_from_request(req)` which now always returns `"dark"`).

**16. ~45 unused `theme = get_theme_from_request(req)` assignments in route modules**.
`app/drafter/routes.py` (11), `app/auth/users.py` (7), `app/docs/routes.py` (5), plus others. Commit intentionally preserves the shim to avoid 40-file churn; track as follow-up cleanup. `app/templates/admin_dashboard.py:150` has the tell-tale `# noqa: F401` comment.

**17. Design-system specs still describe light/system theming** — `docs/superpowers/specs/2026-04-09-design-system.md:127-148` and `…/phase4-design.md:242` describe `[data-theme="dark"]` block, `_THEME_CYCLE_JS`, and `ThemeToggle(theme)`. Update the canonical spec to reflect dark-only.

## Test coverage gaps

- **#655 (Peata on hung retrieval):** no test cancels a turn while `retriever.retrieve` is stuck. Existing cancel test (`test_chat_orchestrator.py:1356`) cancels inside a SlowLLM `astream`, which had yield points even pre-fix. Add a `_HangingRetriever`-based cancellation test.
- **#652 ordering:** the pre-RAG user-message `INSERT` is asserted by substring (`test_chat_orchestrator.py:1684-1691, 1708-1712`) but not by ordering relative to `retrieve()`. Record a timestamp inside `_HangingRetriever.retrieve` and assert user-insert happens first.
- **#656 audit log:** no test asserts `log_action("draft.retry", ...)` fires (`app/docs/retry_handler.py:170`). State-mutating endpoint that re-spends LLM budget — worth pinning.
- **#656 non-HTMX path:** only exercised by the noop test; add a plain-POST happy path (`retry_handler.py:189`).
- **Migration 021:** tests grep the `.sql` for tokens (`test_docs_routes.py:2315-2395`) but never execute the UPDATE. Syntactic drift (misplaced `WHERE`) would still pass. Acceptable per class docstring; flag for integration suite.
- **Warning-count assertion is brittle** — `test_chat_orchestrator.py:1659` pins `len(warning_events) == 1`; future unrelated warnings will flake. Prefer `any(... "aegus" ...)`.
- **#653 slash palette:** JS-only, no regression-guard possible server-side.
